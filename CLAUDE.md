# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Self-hosted RAG chatbot over 363 episodes (10 years) of The Grey NATO podcast. Transcripts with speaker attribution, synopses, and 7,663 curated episode links form the knowledge base. The goal is maximum client-side execution: the browser does vector search via WASM, LM Studio handles only embedding and generation.

## Architecture

```
data/inputs/{n}/episode.md   -->  ingest pipeline  -->  build/podcast.{model}.db
                                                              |
                                                     static file served to browser
                                                              |
                                              browser loads via sql.js + sqlite-vec WASM
                                                              |
                                          user query -> LM Studio /v1/embeddings -> vector search
                                                              |
                                          retrieved chunks -> LM Studio /v1/chat/completions -> streamed response
```

**Client-side (browser via WASM + static files):**
- SQLite with vector search: `sql.js` + `sqlite-vec` (both have WASM builds)
- Entire chunked corpus + pre-computed embeddings as a static `.db` file
- All UI, retrieval, and chat orchestration logic

**Server-side (LM Studio on Mac Studio, localhost:1234):**
- Query embedding via `/v1/embeddings` (convert user question to vector)
- LLM generation via `/v1/chat/completions` (generate streamed response from retrieved context)
- Caddy reverse-proxies `/v1/*` to LM Studio so LAN clients work without LM Studio binding `0.0.0.0`

## Data Format

363 episodes in `data/inputs/{episode_number}/`. Each episode directory contains:

- **`episode.md`** — Primary source. Contains YAML frontmatter, synopsis, links section, and full transcript as a pipe-delimited table:
  ```
  |*Speaker*||
  |----|----|
  |James Stacy| transcript text... |
  |Jason Heaton| transcript text... |
  ```
- **`episode.json`** — Metadata: number, title, mp3_url, episode_url, pub_date
- **`speaker-map.json`** — Maps WhisperX labels (`SPEAKER_00`) to real names
- **`synopsis.txt`** — AI-generated episode summary (standalone copy)
- **`episode-transcribed.json`** — Full transcription with word-level timestamps and speaker labels (~1.4MB each)
- **`whisper-output.json`**, **`whisperx.json`**, **`openai.json`** — Raw transcription outputs (not all present in every episode)

Aggregated data in `data/inputs/sites/tgn/docs/shownotes.md` — all 7,663 links across 359 episodes.

The rendered site lives in `data/site/{episode_number}/episode/index.html` (generated by Zensical).

`episode.md` is the single source file the ingest pipeline needs — it contains synopsis, links, and full attributed transcript.

## Chunking Strategy

Three chunk types, all parsed from `episode.md`:

1. **Synopsis** — One per episode. The synopsis paragraph(s). Good for "which episode discussed X?" queries.
2. **Transcript** — Groups of consecutive speaker turns targeting ~500 words. Preserves conversational context and speaker attribution. Speaker-turn boundaries are natural break points.
3. **Links** — Episode links with anchor text, grouped per episode. For "what was that website about X?" queries.

Each chunk carries metadata: episode number, title, pub_date, chunk type, speakers involved.

## Embedding Models

Build pipeline is model-agnostic. Generates separate `.db` files per model for comparison:

```
build/
  podcast.bge-m3.db
  podcast.nomic-embed-text.db
  podcast.all-minilm.db
```

Candidate models (all loadable in LM Studio via Hugging Face):
- **`bge-m3`** — Dense + sparse retrieval, good for proper nouns (watch names, people)
- **`nomic-embed-text`** — 768-dim, strong general-purpose
- **`all-minilm`** — 384-dim, smallest/fastest, good baseline

Eval script (`ingest/eval.py`) runs test queries with known-good episode matches to compare recall across models.

## SQLite Schema

```sql
CREATE TABLE episodes (
    id INTEGER PRIMARY KEY,
    number INTEGER UNIQUE NOT NULL,
    title TEXT NOT NULL,
    pub_date TEXT,
    synopsis TEXT,
    mp3_url TEXT,
    episode_url TEXT
);

CREATE TABLE chunks (
    id INTEGER PRIMARY KEY,
    episode_id INTEGER NOT NULL REFERENCES episodes(id),
    chunk_type TEXT NOT NULL,  -- 'synopsis', 'transcript', 'links'
    content TEXT NOT NULL,
    speakers TEXT,             -- comma-separated speaker names
    start_turn INTEGER,        -- for transcript chunks
    end_turn INTEGER,
    metadata TEXT              -- JSON blob for extra fields
);

-- sqlite-vec virtual table for vector search
CREATE VIRTUAL TABLE chunks_vec USING vec0(
    chunk_id INTEGER PRIMARY KEY,
    embedding FLOAT[{dim}]     -- dimension depends on model
);
```

## Estimated Sizing

- 363 episodes -> 8,105 chunks (353 synopses, 353 link chunks, 7,399 transcript chunks)
- Embedding size depends on model: 384-dim (all-minilm) to 1024-dim (bge-m3)
- Estimated DB size: **50-150MB** — feasible for browser download; cache in OPFS or Cache API for repeat visits

## Commands

```bash
# Install Python dependencies
pip install -r ingest/requirements.txt

# Run ingest pipeline (requires LM Studio server running locally on :1234)
python ingest/chunk.py                           # parse all episodes -> chunks JSON
python ingest/embed.py --model bge-m3            # embed chunks via LM Studio
python ingest/build_db.py --model bge-m3         # build SQLite DB
python ingest/eval.py                            # compare models

# Serve locally for development
caddy run                                        # or: python -m http.server -d web 8080

# On Mac Studio (production)
caddy start                                      # daemonized, serves on :8080
```

## Key Design Decisions

- **No build step for frontend.** Vanilla JS, no bundler, no node_modules. `<script>` tags only.
- **water.css** for styling — classless CSS, just write semantic HTML.
- **`marked`** (~7KB) for rendering markdown in chat responses.
- **Relative API URLs** (`/v1/chat/completions`, `/v1/embeddings`) — works from any LAN device, no CORS issues, no hardcoded hosts.
- **Streaming responses** via `fetch()` + `ReadableStream`, parsing LM Studio's OpenAI-compatible SSE (`data: {...}\n\n` lines with `choices[0].delta.content`).
- **`episode.md` is the single source of truth** for the ingest pipeline. Other files are supplementary.
- **The `data/` directory is not in git.** It's ~1GB and contains only source data. Manually copy to the Mac Studio.
